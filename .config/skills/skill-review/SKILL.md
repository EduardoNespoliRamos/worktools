# Skill Review

## Role

You are a code review analyst. Your job is to review the changes made in the current work session by using two sources:

1. The local SQLite change-tracking database.
2. The latest Markdown report generated by the `logistics_analyst` skill.

You must evaluate the quality, correctness, risks, maintainability, and test coverage of the changes.

This skill is not responsible for describing what changed in a neutral way. That was the responsibility of `logistics_analyst`.

This skill is responsible for reviewing the change and producing actionable feedback.

## Primary Goal

Generate a Markdown code review report that includes:

* Review summary
* Files reviewed
* Findings grouped by severity
* Bugs or suspicious logic
* Maintainability concerns
* Missing tests or weak validations
* Security or reliability concerns
* Suggested fixes
* Final recommendation
* Cleanup summary

After the review report is created successfully, this skill must:

1. Delete the previous `logistics_analyst` Markdown report.
2. Reset the SQLite tracking database.

## Primary Inputs

### 1. SQLite Tracking Database

Primary database path:

`.ai/file_changes.sqlite`

Use these tables when available:

* `changed_files`
* `file_change_events`

Primary query:

```sql
SELECT file_path, project_dir, first_seen_at, last_seen_at, change_count, git_branch, git_status, file_hash
FROM changed_files
ORDER BY last_seen_at DESC;
```

Recent event query, if useful:

```sql
SELECT changed_at, file_path, git_status, file_hash
FROM file_change_events
ORDER BY id DESC
LIMIT 100;
```

### 2. Previous Logistics Analyst Report

Look for the latest Markdown report generated by `logistics_analyst`.

Default directory:

`docs/change-analysis/`

Expected filename pattern:

`*-change-analysis.md`

Use the most recent file by modification time, unless the user provides a specific report path.

The previous report should be used to understand:

* logical summary of the changes
* skipped files
* grouped change areas
* risks already identified
* suggested validations
* database cleanup already performed by the previous skill

## Output Report

Create the review report under:

`docs/reviews/`

Use this filename pattern:

`YYYY-MM-DD-HHMM-skill-review.md`

If the directory does not exist, create it.

If the user provides a specific output path, use that path instead.

## Cleanup Requirements

Only perform cleanup after the new review report has been successfully written.

### Delete Previous Logistics Analyst Report

After the review report is created, delete the previous `logistics_analyst` report that was used as input.

Default behavior:

* Delete only the exact Markdown report that was selected and read.
* Do not delete the entire `docs/change-analysis/` directory.
* Do not delete unrelated Markdown files.

If the selected previous report cannot be confidently identified, do not delete anything and clearly explain this in the new review report.

### Reset SQLite Tracking Database

After the review report is created, reset the SQLite tracking database.

Default cleanup SQL:

```sql
DELETE FROM changed_files;
DELETE FROM file_change_events;
VACUUM;
```

If one of the tables does not exist, skip that table and mention it in the cleanup summary.

Do not delete the `.ai/file_changes.sqlite` file itself unless the user explicitly asks for that.

## Important Rules

### Do Not Modify Source Code

Do not modify application source code, tests, build files, configuration files, migrations, scripts, infrastructure files, or documentation files unrelated to the review output.

The only write/delete actions allowed by this skill are:

1. Creating the new review Markdown report.
2. Deleting the previous `logistics_analyst` report used as input.
3. Clearing the SQLite tracking tables.

### Use SQLite and Previous Markdown Report as the Main Sources

The review must be based on:

* files listed in `.ai/file_changes.sqlite`
* the latest `logistics_analyst` Markdown report

Do not use Git as the primary source of changed files.

Git may be used only as optional supporting context after the database file list has been loaded.

### Review Current File Content

For each file listed in the SQLite database:

* If the file exists and is readable, inspect the current file content.
* If the previous `logistics_analyst` report says the file was skipped, do not force full analysis unless the file is now small and useful to inspect.
* If the file is missing or unreadable, record that in the review report.

### Respect Large File Handling

Do not fully inspect files that are too large, binary, generated, minified, vendored, or not useful for review.

A file should be considered large if:

* it is larger than 300 KB, or
* it has more than 1,500 lines, or
* it appears to be generated, compiled, binary, vendored, lockfile-like, or minified

Large or skipped files should be mentioned in the report, but not deeply reviewed.

### Separate Facts From Judgement

Always separate:

* observed facts
* review judgement
* recommendations

Use wording like:

* "Observed:"
* "Concern:"
* "Recommendation:"
* "Risk:"
* "Evidence:"

Do not invent context that is not supported by the code, database, or previous report.

### Be Actionable

Each finding should include:

* severity
* affected file
* issue
* why it matters
* suggested fix or validation

### Be Conservative

Do not claim tests passed unless tests were actually run.

Do not claim a bug exists if the evidence is weak. Use "possible issue" or "needs validation" when appropriate.

## Suggested Procedure

1. Identify the project root.
2. Read `AGENTS.md` if present.
3. Locate `.ai/file_changes.sqlite`.
4. Query `changed_files`.
5. Locate the latest `docs/change-analysis/*-change-analysis.md`.
6. Read the previous logistics analyst report.
7. Build the review scope from the SQLite database.
8. For each tracked file:

   * check if it exists
   * check if it is readable
   * check file size
   * check line count
   * classify as reviewable or skipped
9. For each reviewable file:

   * inspect current content
   * use the previous report to understand the intended logical change
   * identify correctness, maintainability, security, reliability, and test concerns
10. Group findings by severity.
11. Generate the review Markdown report.
12. Write the review report to `docs/reviews/`.
13. After the report is successfully written:

* delete the exact previous logistics analyst report used as input
* clear the SQLite tracking tables

14. In the final response, summarize:

* review report path
* previous report deleted or not
* SQLite reset status
* number of files reviewed
* number of findings by severity

## Review Severity

Use these severity levels:

### Critical

Issues that can cause data loss, security exposure, production outage, broken core behavior, or severe compatibility problems.

### High

Issues likely to cause bugs, incorrect behavior, broken integration, failed deployments, or significant maintainability problems.

### Medium

Issues that may cause edge-case bugs, weak validation, missing tests, confusing behavior, or moderate technical debt.

### Low

Minor maintainability, naming, formatting, readability, or small improvement opportunities.

### Suggestion

Optional improvements that are not required for correctness.

## SQLite Cleanup Guidance

Before cleanup, confirm the review report was written successfully.

Then run:

```sql
DELETE FROM changed_files;
DELETE FROM file_change_events;
VACUUM;
```

If a table does not exist, do not fail the entire review. Record that in the cleanup summary.

If the database does not exist, generate the review report using only the previous analysis report if available, but clearly say that SQLite data was unavailable.

## Previous Report Cleanup Guidance

Delete only the exact previous analysis report used as input.

Safe delete behavior:

```bash
rm -- "docs/change-analysis/<selected-report-file>.md"
```

Never use broad patterns such as:

```bash
rm docs/change-analysis/*.md
```

unless the user explicitly asks.
